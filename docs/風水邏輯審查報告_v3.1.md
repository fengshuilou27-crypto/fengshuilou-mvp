# 風水樓 MVP 風水算法邏輯審查報告

**審查日期：** 2026年6月  
**審查範圍：** 9個核心算法模組（data + models）  
**審查標準：** 傳統玄空飛星、八宅、八字命理理論  
**審查者：** 資深風水學專家視角

---

## 一、總體評級：C（中等偏下）

**評級說明：**

- 系統架構設計合理，模組化分工清晰，評分聚合邏輯有思考深度（v2.5 的煞氣防禦分、元運適配係數等設計值得肯定）。
- 但**核心數據層存在根本性錯誤**，尤其是玄空飛星盤的硬編碼數據，大量山盤與向盤完全相同（理論上不可能），中心數字錯誤，格局判定錯誤，這會直接導致用戶得到錯誤的風水評估結果。
- 八字月柱計算存在公式級 Bug，影響所有用戶的命盤準確性。
- 作為 MVP 尚可運行，但**若作為商業產品對外提供風水建議，數據錯誤可能誤導用戶，風險極高**。

**適用場景：** 概念驗證、內部測試、搭配「僅供參考」免責聲明使用。  
**不適用場景：** 收費服務、專業風水師替代、重大購房決策依據。

## 二、各模組詳細審查

### 模組 1：data/flying_star.py（玄空飛星數據層）

**嚴重程度：Critical**

#### 發現問題：

1. **山盤與向盤數據完全相同的根本性錯誤（多處）**
   - 八運 `子山午向`、`丑山未向`、`卯山酉向`、`酉山卯向`、`乙山辛向`、`辛山乙向` 等，其 `mountain_stars` 與 `facing_stars` 完全相同。
   - **風水理論：** 在玄空飛星中，山盤（山星盤）與向盤（向星盤）的排法基於不同的「入中」數字和飛布方向（順飛/逆飛），**任何情況下都不應完全相同**。即使「到山到向」格局，也只是山星到坐山、向星到向首，其他宮位依然不同。
   - **影響：** 這意味著這些數據是編造或複製錯誤的，基於此數據的飛星分析完全不可靠。

2. **中宮數字錯誤（多處）**
   - 八運（運星=8）的 `子山午向` 中宮顯示為 7，`丑山未向` 中宮顯示為 9，`卯山酉向` 中宮顯示為 6。
   - **風水理論：** 玄空飛星中，運盤的中宮數字就是「運星」本身。八運的中宮必須是 8，九運必須是 9，七運必須是 7。這是飛星排盤的起點，錯誤則全盤皆錯。

3. **格局判定錯誤**
   - 八運 `子山午向` 被標為「到山到向」。**理論上八運的旺山旺向為：丑山未向、未山丑向、乾山巽向、巽山乾向、亥山巳向、巳山亥向**。子山午向在八運並非到山到向格局。
   - 九運 `乾山巽向` 被標為「上山下水」。實際上九運到山到向包括子山午向、午山子向等，乾山巽向在九運的準確格局需計算確認，但直接標為「上山下水」需驗證。

4. **硬編碼查表法不可擴展**
   - 僅收錄七、八、九運共 3 個運的數據，且七運數據嚴重缺失（僅 10 個山向）。
   - 玄空飛星的核心是「算法排盤」，而非「查表」。正確做法應根據運數、坐向、陰陽、順逆飛布算法動態生成飛星盤。硬編碼不僅維護困難，且極易出錯。

5. **九運 `丑山未向` 等盤的數據可疑**
   - 九運 `丑山未向`、`未山丑向`、`亥山巳向`、`巳山亥向` 的山盤與向盤完全相同，且中宮數字為 1 或 6（九運應為 9）。這些數據顯然是從八運或其他運數複製過來而未修正。

6. **飛星組合表有遺漏**
   - `FLYING_STAR_COMBO_AUSPICIOUS` 和 `FLYING_STAR_COMBO_INAUSPICIOUS` 缺少一些重要組合，如：
     - 吉：「八一」、「二八」、「三九」等運星組合
     - 凶：「九五」、「九七回祿」、「二六」等
   - 靜態的 `STAR_AUSPICIOUSNESS` 表（「在當運時」）過度簡化，實際星曜吉凶高度依賴當運/失運、組合、宮位。

#### 改進建議：

1. **立即廢除硬編碼飛星盤，改用算法排盤**。參考標準玄空飛星排盤算法：
   - 步驟 1：根據運數定運星（入中宮）
   - 步驟 2：按洛書順飛布運盤
   - 步驟 3：根據坐山（24山）定山星入中宮數字，按陰陽定順逆飛布山盤
   - 步驟 4：根據向首（24山）定向星入中宮數字，按陰陽定順逆飛布向盤
   - 參考《沈氏玄空學》或現代排盤算法實現。

2. 若短期內無法實現算法排盤，至少應請專業風水師逐一核對現有硬編碼數據，修正所有中宮數字和山向盤差異。

3. 補充一運至六運的飛星數據（或算法），支持 180 年完整周期。

4. 增加「反吟」、「伏吟」、「連珠」、「父母三般卦」等進階格局判定。

### 模組 2：data/bazi.py（八字四柱數據層）

**嚴重程度：High**

#### 發現問題：

1. **月柱天干計算公式 Bug（嚴重）**
   ```python
   gan_idx = (start_gan_idx + zhi_idx) % 10
   ```
   - `zhi_idx` 是地支在 `ZHI_LIST` 中的索引（寅=2, 卯=3, ...）。
   - 但正月（寅月）的偏移應為 0，而非 2。
   - **結果：** 甲年正月應為「丙寅」，代碼算出「戊寅」；乙年正月應為「戊寅」，代碼算出「庚寅」。**所有月份的月干都錯了 2 位**（丑月除外，因剛好偏移為 1）。
   - **正確公式：** `gan_idx = (start_gan_idx + (zhi_idx - 2)) % 10`

2. **月柱地支使用公曆月近似**
   - 傳統八字以「節氣」為界（立春為正月、驚蟄為二月等），而非公曆月份。
   - 代碼註釋已承認是「MVP 簡化」，但這會導致約 15% 的用戶（出生在月初/節氣前後）月柱錯誤。
   - **建議：** 至少實現「節氣查表法」，將 24 節氣日期硬編碼，根據出生日期判斷月支。

3. **日柱基準日未經驗證**
   - 代碼假設 `2000-01-01 = 己卯日`。若此基準錯誤，所有日柱計算將系統性偏移。
   - 建議使用萬年曆數據庫或權威曆法資料驗證此基準。

4. **時柱計算的小 Bug**
   - `TIME_TO_ZHI` 中同時存在 `(0, 1): "子"` 和 `(23, 24): "子"`，但 `get_hour_zhi` 的優先邏輯已經覆蓋了 23:00-01:00。這是冗余代碼，無實際影響。

5. **日主旺衰計算過度簡化**
   - 僅考慮：月令（40%）+ 通根（30%）+ 天干生扶（30%）。
   - **缺失關鍵因素：**
     - 地支藏干的「本氣/中氣/餘氣」強度差異（如丑中己土為本氣、癸為中氣、辛為餘氣，應有不同權重）
     - 天干合化（如甲己合化土、乙庚合化金）
     - 空亡（旬空）對旺衰的影響
     - 十神格局（如正官格、七殺格等）對喜用神的影響
     - 大運、流年對當前旺衰的動態影響
   - 旺衰判定閾值過於粗糙（≥5 為旺，≤-5 為弱），實際命理中「中和」與「偏旺/偏弱」的判定需更精細。

6. **喜用神計算不完整**
   - 僅簡單地：旺→取官殺/食傷，弱→取印星/比劫，中和→取印星/官殺。
   - 實際命理中，喜用神需綜合考慮：調候（寒暖燥濕）、通關（兩神相戰取中間）、從格（從強/從弱）等特殊情況。系統完全未考慮「從格」，這是初學者常犯的錯誤。

7. **樓層納甲系統有創意但非標準**
   - 以建築坐向定「本宅卦」，按後天八卦序循環定各樓層五行。這是一個現代創新，但並非傳統風水理論的標準做法。
   - 傳統樓層五行通常只用「河圖數」（1、6 水，2、7 火，3、8 木，4、9 金，5、0 土）。
   - 建議：標註為「創新算法」而非「傳統納甲」，避免誤導用戶。

8. **職業五行映射部分不合理**
   - 「傳媒/廣告/公關」= 水（合理，信息流動）
   - 「科技/IT/電子」= 火（合理，電屬火）
   - 但「管理/顧問/策劃」= 土（土主信、主靜，管理更偏向火/金的統籌與決斷，此歸類偏牽強）
   - 建議增加更多職業，並允許用戶自選五行。

9. **樓層匹配邏輯有不可達代碼**
   ```python
   if floor_wuxing == yong_shen:
       return {...}  # 第 1 個條件
   # ... 中間條件 ...
   if floor_wuxing == yong_shen:
       return {...}  # 第 6 個條件，永遠不可達
   ```
   最後一個 `if floor_wuxing == yong_shen` 是死代碼，雖無實際影響，但反映代碼審查不嚴謹。

#### 改進建議：

1. **緊急修復月干公式**：`gan_idx = (start_gan_idx + (zhi_idx - 2)) % 10`
2. 實現節氣查表（至少硬編碼未來 50 年的立春、驚蟄等關鍵節氣日期）。
3. 驗證 `2000-01-01` 的日柱是否確為「己卯」。
4. 增加「從格」判定邏輯（日主極旺或極弱時的 special handling）。
5. 增加「調候用神」計算（如冬生者需火調候，夏生者需水調候）。
6. 將納甲樓層算法標註為「現代創新算法」。

### 模組 3：models/flying_star_analysis.py（飛星分析模組）

**嚴重程度：High**

#### 發現問題：

1. **完全依賴底層錯誤數據**
   - 由於 `data/flying_star.py` 的飛星盤數據錯誤，此模組的分析結果必然錯誤。這是「垃圾進，垃圾出」的典例。

2. **七運數據回退機制不合理**
   ```python
   elif yun == "七運":
       if "八運" in FLYING_STAR_TABLE and building_facing in FLYING_STAR_TABLE["八運"]:
           chart = FLYING_STAR_TABLE["八運"][building_facing]
   ```
   - 七運與八運的飛星盤完全不同（運星不同、入中數字不同、順逆可能不同），用八運數據回退七運查詢是**嚴重錯誤**。
   - 寧可返回「數據缺失」也不要用錯誤數據回退。

3. **吉凶組合計分過於簡單**
   - `auspicious_score = len(auspicious_combos) * 5` — 僅統計數量，不考慮組合所在的宮位重要性（如大門、臥室、廚房 vs 儲物間）。
   - 同樣的「88」組合在財位（向首）和廁所位，吉凶影響完全不同。

4. **annual_overlay 未實際填充**
   - 代碼支持 `annual_overlay`（年度疊加星），但數據層中幾乎所有飛星盤的 `annual_overlay` 都為空。此功能形同虛設。

5. **異運加權 70%/30% 缺乏理論依據**
   - 建造運 70% + 當運 30% 的加權是工程師的假設，非傳統風水理論。
   - 傳統上，元運轉換後的樓宇風水評估需考慮「換天心」與否。若未換天心，建造運盤仍主導；若已換天心，當運盤主導。不能簡單用固定權重。

#### 改進建議：

1. 待底層數據修復後，此模組需重新測試所有路徑。
2. 移除七運→八運回退機制，改為返回「暫不支持該運數/坐向」。
3. 增加宮位重要性加權（大門、臥室、廚房、廁所等應有差異化評分）。
4. 實現年度流年飛星疊加功能（每年流年星入中宮，與宅運盤疊加分析）。
5. 異運分析增加「是否已換天心」的用戶輸入選項，動態調整權重。

### 模組 4：models/bazi_matching.py（八字匹配模組）

**嚴重程度：Medium**

#### 發現問題：

1. **樓層匹配僅基於喜用神，過度簡化**
   - 傳統風水中，樓層選擇與八字的關係是多維度的：不僅要考慮喜用神，還要考慮生肖（屬相納音）、命卦、五子運等。
   - 系統僅用「喜用神五行 = 樓層五行」判定，忽略了「日主強弱極端時不宜直接用喜用神」的情況（如從強格忌印星）。

2. **職業加成與喜用神的耦合邏輯有風險**
   - 若 `wang_shuai` 計算錯誤（見模組 2），則職業加成也會錯誤。錯誤會級聯傳播。

3. **缺少八字與建築坐向的直接匹配**
   - 傳統風水中有「命卦配宅卦」的邏輯（見八宅模組），但八字模組未考慮「日主五行與宅向五行的生克」。例如，日主屬水的人住坐南朝北（離火宅）可能有壓制。

#### 改進建議：

1. 增加「生肖納音」與樓層的匹配（如甲子年屬金，宜住 4、9 樓）。
2. 增加「五子運」判斷（甲子運、丙子運等），用於樓宇與運勢的匹配。
3. 增加日主與宅向五行的直接生克分析。
4. 在喜用神計算不穩定時，增加「保守模式」——僅用日主五行做粗略匹配，降低風險。

### 模組 5：models/bagua_matching.py（八宅匹配模組）

**嚴重程度：Medium**

#### 發現問題：

1. **命卦計算使用公曆年，未考慮農曆年**
   - `calc_ming_gua` 直接用 `birth_year`（公曆年份）。
   - **風水理論：** 命卦計算應使用「農曆年份」。若出生日期在立春前（如 1990 年 1 月 15 日），農曆年仍為 1989 年（己巳年），命卦應按 1989 計算，而非 1990。
   - 影響：每年約 15% 的用戶（1-2 月出生）命卦可能錯誤。

2. **東四命/西四命匹配分數偏低**
   - 宅命相配僅佔 4 分（滿分 10 分），而房間九宮分析佔 6 分。傳統八宅理論中，「宅命相配」是首要條件，不匹配則其他調整效果有限。建議提高宅命匹配權重。

3. **八宅吉凶方位表正確**
   - `BAGUA_DIRECTION_TABLE` 的八個卦的吉凶方位排列是標準的，此部分無誤。

4. **房間重要性權重合理**
   - `ROOM_IMPORTANCE` 中「大門 3.0、臥室 2.5、客廳 2.0」的設定符合八宅理論。

5. **雙人版衝突檢測邏輯正確**
   - A 的吉位是否為 B 的凶位，這是雙人居住的實際風水問題，檢測邏輯正確。

#### 改進建議：

1. 增加農曆年轉換：根據出生日期與當年立春日期比較，確定命卦計算用年。
2. 提高宅命相配權重（建議佔 50% 以上），或增加「宅命不配時的化解建議」（如改睡房到伏位/天醫位）。
3. 增加「大門朝向與命卦的關係」分析（大門宜開在生氣、延年、天醫位）。

### 模組 6：models/zero_main_god.py（零正神分析模組）

**嚴重程度：Medium**

#### 發現問題：

1. **零正神方位表基本正確**
   - 八運：正神東北、零神西南 ✓
   - 九運：正神南方、零神北方 ✓
   - 七運：正神西方、零神東方 ✓
   - 此部分數據正確。

2. **回退邏輯的方位映射過於寬鬆**
   - 八運回退中：「南有山」映射為「西南有山（零神見山）」，「北有水」映射為「東北有水（正神見水）」。
   - 南 ≠ 西南，北 ≠ 東北。這種映射誤差可達 45 度，在風水羅盤中屬於「出卦」，可能導致完全不同的吉凶判斷。
   - 建議：若無精確方位數據，應直接標註「數據不足，無法判定」，而非用寬鬆映射。

3. **缺少「正神見山/零神見水」的量化判定**
   - 系統僅有「有/無」的布爾值，但實際風水中：
     - 山的高低、遠近、形狀影響「見山」的效力
     - 水的大小、動靜、清濁影響「見水」的效力
   - 作為 MVP，「有/無」簡化可接受，但應在文檔中說明。

#### 改進建議：

1. 移除寬鬆方位映射，無精確數據時降低置信度並提示用戶。
2. 增加「正神零神」與「飛星盤動態對應」分析：例如，即使正神在東北宜見山，若東北飛星盤出現「二五交加」，則「見山」可能反而加劇病符。
3. 增加「零正神」與「催財水法」的聯動建議（如零神方設水景、魚缸）。

### 模組 7：models/sha_assessment.py（煞氣評估模組）

**嚴重程度：Medium**

#### 發現問題：

1. **v2.5 的「煞氣防禦分」設計是亮點**
   - 將煞氣從「負分扣分」改為「正向防禦分」，避免了負分截斷問題，架構設計值得肯定。

2. **煞氣種類嚴重不足**
   - `SHA_TABLE` 僅收錄 6 種煞氣（天斬煞、聲煞、路沖、二五交加、伏吟、其他）。
   - **實際風水中常見的煞氣至少 20-30 種**，缺失包括：
     - 反弓煞（反弓水/反弓路）
     - 穿心煞（電梯/樓梯直沖大門）
     - 剪刀煞（Y 字路口）
     - 蜈蚣煞（多條管道/電線）
     - 白虎煞（右方高壓/尖角）
     - 探頭煞（前方建築凸出）
     - 孤峰煞（獨高無靠）
     - 廉貞煞（獨棟高樓）
     - 陰煞（墳場、醫院、垃圾站附近）
     - 光煞（強光反射）
     - 電磁煞（高壓電塔、變電站）

3. **飛星盤煞氣推導有局限**
   - `derive_sha_from_pan` 僅檢查「二五交加」、「六七交劍」、「五黃到宮」、「三碧是非」、「伏吟」。
   - 缺少：「反吟」（山星與向星對宮相沖，如 1 對 9、2 對 8 等）、「九七回祿」（火克金）、「土煞」（五二/二五之外的三五、五八等）。
   - 伏吟判定僅檢查 [2, 5, 7]，實際上任何星數的伏吟（山星=向星同宮）都是氣滯之象，不限於這三個數。

4. **化解建議過於簡單**
   - 僅給出「麒麟一對」、「山海鎮」等通用建議，未根據煞氣類型和方位給出具體擺放位置（如「天斬煞：麒麟放窗台面向煞方」）。

#### 改進建議：

1. 擴充煞氣數據庫至至少 20 種常見煞氣，每種包含：名稱、描述、嚴重程度、扣分、化解方法、化解物品、擺放方位、參考成本。
2. 增加「反吟」檢測：山星與向星在對宮相沖（如北 1 vs 南 9、東北 2 vs 西南 8 等）。
3. 增加「九七回祿」檢測（火克金，火災之險）。
4. 根據煞氣類型，給出具體的化解擺放方位（如「路沖：山海鎮應懸掛於大門外正上方，面向來路」）。

### 模組 8：models/goal_matching.py（目標匹配模組）

**嚴重程度：Low-Medium**

#### 發現問題：

1. **目標對應的吉星映射過度簡化**
   - `GOAL_STAR_TABLE` 中，每個運的每個目標僅對應 2-3 顆星。
   - 例如「桃花」在八運對應「一白、九紫」。實際上，桃花還與四綠（文昌/桃花）、卯位（震宮）、以及流年桃花位（如 2026 丙午年桃花在卯）相關。
   - 缺少流年因素（當前年份的桃花位、財位等）。

2. **目標對應的組合表有創意但需驗證**
   - `GOAL_KEY_COMBOS` 和 `GOAL_AVOID_COMBOS` 是作者根據經驗整理的，整體方向合理，但：
     - 「桃花」吉組合包含「49」、「94」——四綠為文昌，九紫為喜慶，組合確實主文藝桃花，但「41」為文昌，與桃花關聯較弱。
     - 「健康」凶組合包含「57」、「75」——五七同宮主破財傷丁，但「75」是否與健康直接相關，需更精確的理論支持。

3. **宅盤類型加成缺乏理論依據**
   - 「財富」目標下，「到山到向」+3，「上山下水」-3。這是合理的（旺財）。
   - 但「桃花」目標下「到山到向」+2，「上山下水」無調整——實際上桃花與宅盤類型的關聯較弱，更多取決於飛星組合和方位。

4. **多目標權重設計合理**
   - 主目標 100%、次目標 50%、第三目標 25% 的加權是合理的 UX 設計，避免目標過多導致分數稀釋。

#### 改進建議：

1. 增加「流年目標星曜」計算：根據評估年份，計算流年財位、文昌位、桃花位，與宅運盤對比。
2. 增加「九紫喜慶星」的流年位置對家庭和睦、桃花目標的影響。
3. 增加「一白貪狼星」的流年位置對事業、桃花目標的影響。
4. 允許用戶自訂目標權重，而非固定 100%/50%/25%。

### 模組 9：models/match_result.py（匹配結果聚合模組）

**嚴重程度：Low**

#### 發現問題：

1. **v2.5 的架構改進值得肯定**
   - 煞氣改防禦分、樓齡改健康分、地段剝離為投資屬性、元運交叉改適配係數——這四項改進顯著提升了評分系統的合理性。

2. **權重分配有討論空間**
   - 飛星 20 + 八字 18 + 八宅 13 + 零正神 8 + 目標 13 + 物業特徵 10 + GIS 8 + 煞氣防禦 7 = 97
   - 飛星權重（20）高於八宅（13），這偏向「玄空派」視角。若用戶是「八宅派」風水師，可能認為八宅權重過低。
   - 建議：在設置中允許用戶選擇「玄空派權重」、「八宅派權重」、「均衡權重」等模式。

3. **元運適配係數範圍過大**
   - `yun_adaptation_factor = 1.0 + (multi_yun_adjust / 50)`，範圍 0.85~1.15。
   - 當 `multi_yun_adjust = -5` 時，係數 = 0.9，總分直接打 9 折。這個懲罰力度偏大。例如一個原本 85 分的樓宇，因元運轉換直接降到 76.5 分，可能過度懲罰。
   - 建議：縮小係數範圍至 0.95~1.05，或改為固定加減分而非乘法係數。

4. **樓層微調（floor_tie_breaker）設計有問題**
   - `floor_tie_breaker = min(floor_number * 0.01, 0.3)`，高樓層加分。
   - 這會導致「在風水條件完全相同的情況下，50 樓比 1 樓高 0.29 分」。這與風水無關，屬於技術層面的「避免分數相同」手段。
   - 建議：用「隨機鹽值」或「基於物業 ID 的哈希值」作為 tie-breaker，而非樓層數。高樓層在風水上不一定更好（如頂樓可能犯「孤峰煞」）。

5. **風險控制計算有數學問題**
   ```python
   family_risk = abs(raw_sha_penalty) + abs(zmg_score) if zmg_score < 0 else abs(raw_sha_penalty)
   ```
   - 若 `zmg_score = -5`，則 `family_risk = 15 + 5 = 20`。但若 `zmg_score = -10`，`family_risk = 15 + 10 = 25`。
   - 這個「家庭風險係數」沒有標準化解釋，用戶難以理解「20」代表什麼。
   - 建議：將風險係數轉換為「低/中/高/極高」的定性標籤，而非原始數值。

6. **評級文字過度保守**
   - 所有評級都帶有「需專業師傅確認」，這雖然是免責策略，但降低了 AI 的實用性。
   - 建議：高分區間（85+）可給予「此單位風水條件較優，建議實地確認」的正面表述，低分區間才強調「需專業確認」。

7. **缺少總分與模組分的交叉驗證**
   - 若某個模組返回錯誤（如八字月干 Bug），總分仍會正常計算，用戶無法感知底層數據錯誤。
   - 建議：增加「數據健康檢查」標記，當某模組置信度低於閾值時，在報告中突出警告。

#### 改進建議：

1. 增加「派別權重」設置（玄空派/八宅派/均衡派）。
2. 將元運適配係數改為加減分（±5 分），避免乘法導致的極端波動。
3. 將樓層微調改為基於物業 ID 的哈希 tie-breaker。
4. 增加「數據健康度」指標，讓用戶知道哪些模組的數據質量高/低。
5. 優化評級文案，區分「推薦」與「警告」語氣。

## 四、需要補充的風水要素

### 1. 流年飛星（年度運勢）
- 當前系統僅分析「宅運盤」（固定格局），缺少每年流年星的疊加分析。
- 例如：2026 年為丙午年，流年五黃入中宮，所有宅運盤的吉凶都會因流年而變化。這是買房者最關心的「今年能不能買」的問題。

### 2. 太歲與歲破
- 每年太歲方位（如 2026 年太歲在午/南方）忌動土、修造。系統未考慮。

### 3. 三煞方
- 每年三煞方位（如申子辰年三煞在南方）忌開門、動土。系統未考慮。

### 4. 五黃與二黑的流年位置
- 五黃大煞和二黑病符每年飛臨不同方位，這是風水實踐中最重要的年度調整依據。系統未實現。

### 5. 羅盤二十四山精確度
- 當前系統僅支持 24 山向，但實際風水羅盤分為 24 山（每山 15°），部分流派使用 72 龍或 120 分金。坐向偏差 1° 可能導致完全不同的飛星盤。
- 建議：增加坐向精確度輸入（如「子山午向兼壬丙」）。

### 6. 陽宅三要（門、主、灶）
- 八宅派核心：大門、主臥、廚房的位置關係。系統有「房間位置」分析，但未強調「門主灶」的互動（如大門在生氣位、臥室在伏位、廚房在五鬼位以穢制穢）。

### 7. 形煞與理氣的結合
- 當前系統將「物理煞氣」（天斬煞）與「理氣煞氣」（二五交加）分開處理。
- 實際風水中，形煞與理氣需結合：如天斬煞若剛好沖向五黃飛臨的方位，凶性加倍；若沖向九紫飛臨的方位，可能「以吉化凶」。

### 8. 玄空大卦（64 卦）
- 進階玄空風水會分析 24 山對應的 64 卦（如子山午向對應 坎為水卦），判斷「父母卦」、「江東卦」、「江西卦」等。
- 此為進階功能，但可作為未來擴展方向。

### 9. 風水擇日
- 買房、入伙、裝修的吉日選擇。系統目前僅做「靜態分析」，缺少「動態擇日」。

### 10. 五子運（樓宇與人的運勢匹配）
- 甲子運（1984-2043）、丙子運等五子運系統，用於判斷樓宇元運與人運的匹配。系統未考慮。
